# Monitoring Storage Migration

Prometheus and Loki move from the `monitoring` namespace to dedicated
application namespaces. Prometheus has static PV/PVC manifests in this repo;
Loki's PVC is generated by the Helm chart.

The repository intentionally leaves `argocd/system/monitoring.yaml` without
automated prune and leaves the new `prometheus`, `loki`, and `otel-external`
Applications without auto-sync. This prevents Argo CD from deleting the old
stateful resources before storage is rebound.

## Sync order

1. Sync `system/monitoring` with prune disabled.
2. Migrate Prometheus and Loki storage as described below.
3. Delete or release the old `monitoring` namespace resources that moved out:
   Prometheus, Loki, and `otelcol-external`.
4. Sync `applications/prometheus`, `applications/loki`, and
   `applications/otel-external`.
5. After the moved apps are healthy, re-enable automated prune/sync if desired.

## Prometheus

Prometheus keeps the existing PV and PVC names:

- PV: `pvc-2a79df8c-7605-4225-9a79-04330e38d000`
- PVC: `prometheus-prometheus-db-prometheus-prometheus-0`

The manifests bind that PV/PVC pair in the `prometheus` namespace.

## Loki

The Loki chart renders a StatefulSet named `loki` with a `storage`
volume claim template, so the PVC name is expected to be `storage-loki-0`.
Before syncing the new `applications/loki` app, rebind the existing Loki PV
from `monitoring/storage-loki-0` to `loki/storage-loki-0`.

Use the live cluster to identify the PV first:

```sh
kubectl get pvc storage-loki-0 -n monitoring -o jsonpath='{.spec.volumeName}{"\n"}'
```

Then patch that PV's claimRef to the new namespace after the old Loki workload
has been stopped and the old PVC has been removed or released:

```sh
kubectl patch pv <loki-pv-name> --type merge -p '{"spec":{"claimRef":{"namespace":"loki","name":"storage-loki-0"}}}'
```

Do not delete the backing PV. The Loki chart still requests a 10Gi
`beryllium-iscsi` ReadWriteOnce volume.
